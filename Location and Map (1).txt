3. Rules
3.1 Maps Display
Web: Google Maps Embed API on business profile pages — free, unlimited.
Mobile: Google Maps SDK — free, unlimited.
No map auto-loads. Default view is always list view. Map loads only when user clicks "View on Map".
Mobile fallback: if SDK fails, show a static OpenStreetMap tile + address text + Retry button (one try/catch in Flutter — no full OSM integration). See Section 3.11 for OSM fair-use rules.

3.2 User Location
Device GPS only — no external API. Accuracy: within 100 m. Add a Refresh button.
If GPS denied or unavailable: Manual city selection → Manual pin placement.
Never request location at app launch.
Never store GPS location beyond the active session.

3.3 Address Input — Business Registration Only
Use Google Places Autocomplete. Trigger after 3+ characters with 300 ms debounce.
Session tokens are mandatory — without them every keystroke is a separate billable API call.
Store only: name, address, city, lat, lng. Do not store any raw Google Places data.

3.4 Geocoding Queue
Never geocode on the frontend. Use BullMQ queue on the backend.
Flow: user submits → profile saved (status: pending_geocode) → BullMQ queues job → worker updates lat/lng → status: active.
Before calling the API, check if the same address already exists in DB — reuse stored coordinates if found.
Retry failed geocoding up to 3 times. Send admin alert if all retries fail.

Rate limit fix: BullMQ worker concurrency to maximum 10 concurrent jobs, with a minimum 100 ms delay between jobs. This prevents hitting Google's QPS limit during registration spikes.

3.5 Search & Nearby — Combined Elasticsearch + PostGIS Pipeline

ARCHITECTURE RULE: Elasticsearch handles text relevance. PostGIS handles geo-boundary filtering. Both are mandatory and must work together in every search request. Never use one without the other for business discovery queries.

Pipeline Flow
Every business discovery search must follow this exact pipeline:

Step
Engine
What it does
1
Elasticsearch
Receives query text + category + filters. Scores and ranks matching business documents by relevance. Returns a list of matching business IDs with scores.
2
PostGIS (ST_DWithin)
Takes the business IDs from Step 1. Applies the user's chosen radius filter (1 km, 5 km, 10 km, 25 km, 50 km). Eliminates results outside the boundary.
3
Merge & Sort
Combines Elasticsearch relevance score + distance. Applies final sort (relevance, distance, or rating as chosen by user). Returns paginated result set.
4
Redis Cache
Caches the final merged result set keyed by search parameters. TTL: 15 minutes. See Section 3.6 for full cache rules.

Elasticsearch Index Requirements
Index name: businesses. Map the following fields: name (text, analyzed), category (keyword), tags / keywords (text), city (keyword), status (keyword), lat_lng (geo_point).
Use a custom analyzer for Urdu/Roman Urdu transliteration where applicable.
Always filter by status: active before returning results — never surface pending or suspended listings.
Boost fields in order: exact name match > category match > keyword match > description match.
Pagination: use search_after (not from/size) for deep pagination at scale.

PostGIS Geo-Filter Requirements
Use ST_DWithin on the geography type (not geometry) for accurate metre-based distance calculations.
Add a GIST spatial index on every location column: CREATE INDEX CONCURRENTLY idx_businesses_location ON businesses USING GIST(location);
Radius options available to user: 1 km, 5 km, 10 km, 25 km, 50 km.
Pass only the business_id list from Elasticsearch into the PostGIS WHERE clause — never run PostGIS on the full table without an ID filter.
Never use Google Nearby Search API, Distance Matrix API, or Directions API.

3.6 Redis Cache — Search Results (Full Rules)

FIX APPLIED v2.4: Full cache invalidation rules added. Missing invalidation causes stale results — closed businesses shown as open, deleted listings appearing in search. This is a trust-destroying bug. Follow all rules below without exception.

Cache Key Structure
All search result cache keys must follow this exact format:

search:{city_slug}:{category_slug}:{radius_km}:{query_hash}

Example: search:lahore:restaurants:5:a3f9c1d2. The query_hash is an MD5 or SHA-1 of the normalized query string + active filters.

TTL Rules
Data Type
TTL
Search results (city + category + radius)
15 minutes
Individual business profile data
30 minutes
Popular search suggestions
1 hour
City / category index lists
6 hours

Invalidation Rules — Mandatory
On business address update: immediately delete all cache keys matching search:{city_slug}:*. Use Redis SCAN + DEL — never KEYS in production.
On business status change (activation, suspension, deletion): immediately purge all cache keys that could contain this business. Fire a BullMQ cache-invalidation job on every status transition.
On business category update: purge cache keys for both old and new categories in the affected city.
On business deletion: purge business profile cache key (business:{id}) AND all city-level search caches where it appeared.
On business hours update (open/closed status change): purge the business profile cache immediately. Search caches with "Open Now" filter must be purged within 1 minute.
Never use FLUSHDB or FLUSHALL for targeted invalidation — these wipe the entire cache.

Cache Warming
After any bulk invalidation event, queue a BullMQ cache-warm job for the top 20 most-searched city/category/radius combinations.
Warm jobs run at off-peak times (02:00–05:00 local) to repopulate expired popular searches.

3.7 Navigation
Add 'Open in Google Maps' button on every business profile.
Link format: https://www.google.com/maps?q=LAT,LNG
No in-app navigation. No Routes API. No Directions API.

3.8 Duplicate Business Detection
Flag as duplicate only when 2 or more of these signals match simultaneously:

Signal
Rule
GPS coordinates
Within 10 m of an existing listing
Business name
80%+ similarity — use pg_trgm. Normalize before comparison: lowercase, trim spaces, strip punctuation, normalize Unicode, apply transliteration.
Phone number
Exact match — normalize to E.164 format before comparison.
Address text
85%+ similarity — use pg_trgm after same normalization as name.

1 signal match = soft warning, user can continue.
2+ signals = confirmation prompt shown to user.
Never hard-block automatically — always allow user override.
Validate on both frontend and backend.

pg_trgm is already available in PostgreSQL with PostGIS setups. It is free, fast, and handles fuzzy text matching natively — no extra library needed. Enable with: CREATE EXTENSION IF NOT EXISTS pg_trgm;

3.9 Provider Abstraction Layer
Never call Google API clients directly in application code.
Wrap all map/location API calls in three NestJS injectable services: GeocoderService, PlacesService, MapProvider.
The rest of the app calls only these services. Switching providers later requires changing only these files.
Add a fourth service: SearchLocationService — this is the single entry point for all combined Elasticsearch + PostGIS search queries. No other layer calls Elasticsearch or PostGIS directly for search.

3.10 Cost Controls
Set hard daily quota limits for all paid APIs in Google Cloud Console.
Set billing alerts at 50% and 90% of monthly budget.

Important: Business profile pages must remain publicly accessible (no login required to view them). The Google Maps Embed API is free for public pages. Putting it behind a login wall may violate Google's Terms of Service.

3.11 OpenStreetMap Tile Fair-Use Policy (Mobile Fallback Only)

NEW in v2.4: OSM tile fair-use rules added. Violating these terms can result in your app being IP-blocked by OSM tile servers — causing the fallback to fail for all users.

The OSM static tile fallback is used only in the mobile app, only when the Google Maps SDK fails, and only for a single static tile per failure event. Despite this limited use, the following rules apply:

Rule
Detail
Tile server
Use tile.openstreetmap.org for MVP only. If monthly active users exceed 50,000, migrate to a self-hosted tile server or a CDN-backed provider (e.g., Stadia Maps free tier, Protomaps).
Usage cap
The OSM Tile Usage Policy prohibits bulk or automated tile fetching. This fallback fires only on SDK failure — one tile per event. This is compliant.
Caching
Cache returned OSM tiles on the device for a minimum of 24 hours. Never re-fetch the same tile within 24 hours.
Attribution
Display "© OpenStreetMap contributors" text on any screen showing an OSM tile. This is legally mandatory under the ODbL licence.
Migration trigger
At 50,000+ MAU: migrate OSM tiles to Stadia Maps free tier (no key required up to 200,000 tile requests/month) or self-hosted. Document this in post-launch roadmap.
