Rules`n`n`n3.1 Maps Display`n`n`nWeb: Google Maps Embed API on business profile pages — free, unlimited.`n`n`nMobile: Google Maps SDK — free, unlimited.`n`n`nNo map auto-loads. Default view is always list view. Map loads only when user clicks "View on Map".`n`n`nMobile fallback: if SDK fails, show a static `nOpenStreetMap`n tile + address text + Retry button (one try/catch in Flutter — no full OSM integration). See Section 3.11 for OSM fair-use rules.`n`n`n`n`n3.2 User Location`n`n`nDevice GPS only — no external API. Accuracy: within 100 m. `nAdd`n a Refresh button.`n`n`nIf GPS denied or unavailable: Manual city selection → Manual pin placement.`n`n`nNever request location at app launch.`n`n`nNever store GPS location beyond the active session.`n`n`n`n`n3.3 Address Input — Business Registration Only`n`n`nUse Google Places Autocomplete. Trigger after 3+ characters with 300 `nms`n `ndebounce`n.`n`n`nSession tokens are mandatory — without them every keystroke is a separate billable API call.`n`n`nStore only: name, address, city, `nlat`n, `n`nlng`n`n. Do not store any raw Google Places data.`n`n`n`n`n3.4 Geocoding Queue`n`n`nNever geocode on the frontend. Use `nBullMQ`n queue on the backend.`n`n`nFlow: user submits → profile saved (status: `npending_geocode`n) → `nBullMQ`n queues job → worker updates `nlat`n/`nlng`n → status: active.`n`n`nBefore calling the API, check if the same address already exists in DB — reuse stored coordinates if found.`n`n`nRetry failed geocoding up to 3 times. Send admin alert if all retries fail.`n`n`n`nRate limit fix: `nBullMQ`n worker concurrency to maximum 10 concurrent jobs, with a minimum 100 `nms`n delay between jobs. This prevents hitting Google's QPS limit during registration spikes.`n`n`n`n`n3.5 Search & Nearby — Combined `nElasticsearch`n + `nPostGIS`n Pipeline`n`n`n`nARCHITECTURE RULE: `nElasticsearch`n handles text relevance. `nPostGIS`n handles geo-boundary filtering. Both are mandatory and must work together in every search request. Never use one without the other for business discovery queries.`n`n`n`n`nPipeline Flow`n`nEvery business discovery search must follow this exact pipeline:`n`n`nStep`nEngine`nWhat it does`n1`n`nElasticsearch`n`nReceives query text + category + filters. Scores and ranks matching business documents by relevance. Returns a list of matching business IDs with scores.`n2`n`nPostGIS`n (`nST_DWithin`n)`nTakes the business IDs from Step 1. Applies the user's chosen radius filter (1 km, 5 km, 10 km, 25 km, `n50`n km). Eliminates results outside the boundary.`n3`nMerge & Sort`nCombines `nElasticsearch`n relevance score + distance. Applies final sort (relevance, distance, or rating as chosen by user). Returns paginated result set.`n4`n`nRedis`n Cache`nCaches the final merged result set keyed by search parameters. TTL: 15 minutes. See Section 3.6 for full cache rules.`n`n`n`n`n`nElasticsearch`n Index Requirements`n`n`nIndex name: businesses. Map the following fields: name (text, analyzed), category (keyword), tags / keywords (text), city (keyword), status (keyword), `nlat_lng`n (`ngeo_point`n).`n`n`nUse a custom analyzer for Urdu/Roman Urdu transliteration where applicable.`n`n`nAlways filter by status: active before returning results — never surface pending or suspended listings.`n`n`nBoost fields in order: exact name match > category match > keyword match > description match.`n`n`nPagination: use `nsearch_after`n (not from/size) for deep pagination at scale.`n`n`n`n`n`nPostGIS`n Geo-Filter Requirements`n`n`nUse `nST_DWithin`n on the geography type (not geometry) for accurate `nmetre`n-based distance calculations.`n`n`nAdd a GIST spatial index on every location column: CREATE INDEX CONCURRENTLY `nidx_businesses_location`n ON businesses USING GIST(location);`n`n`nRadius options available to user: 1 km, 5 km, 10 km, 25 km, 50 km.`n`n`nPass only the `nbusiness_id`n list from `nElasticsearch`n into the `nPostGIS`n WHERE clause — never run `nPostGIS`n on the full table without an ID filter.`n`n`nNever use Google `nNearby`n Search API, Distance Matrix API, or Directions API.`n`n`n`n`n3.6 `nRedis`n Cache — Search Results (Full Rules)`n`n`n`nFIX APPLIED v2.4: Full cache invalidation rules added. Missing invalidation causes stale results — closed businesses shown as open, deleted listings appearing in search. This is a trust-destroying bug. Follow all rules below without exception.`n`n`n`n`nCache Key Structure`n`nAll search result cache keys must follow this exact format:`n`n`nsearch:{`ncity_slug`n}:{`ncategory_slug`n}:{`nradius_km`n}:{`nquery_hash`n}`n`n`n`nExample: search`n:lahore:restaurants:5:a3f9c1d2`n. The `nquery_hash`n is an MD5 or SHA-1 of the normalized query string + active filters.`n`n`n`n`nTTL Rules`n`nData Type`n`nTTL`n`nSearch results (city + category + radius)`n`n15 minutes`n`nIndividual business profile data`n`n30 minutes`n`nPopular search suggestions`n`n1 hour`n`nCity / category index lists`n`n6 hours`n`n`n`n`nInvalidation Rules — Mandatory`n`n`nOn business address update: immediately delete all cache keys matching `nsearch:`n{`ncity_slug`n}:*. Use `nRedis`n SCAN + DEL — never KEYS in production.`n`n`nOn business status change (activation, suspension, deletion): immediately purge all cache keys that could contain this business. Fire a `nBullMQ`n cache-invalidation job on every status transition.`n`n`nOn business category update: purge cache keys for both old and new categories in the affected city.`n`n`nOn business deletion: purge business profile cache key (`nbusiness:`n{id}) AND all city-level search caches where it appeared.`n`n`nOn business hours update (open/closed status change): purge the business profile cache immediately. Search caches with "Open Now" filter must be purged within 1 minute.`n`n`nNever use FLUSHDB or FLUSHALL for targeted invalidation — these wipe the entire cache.`n`n`n`n`nCache Warming`n`n`nAfter any bulk invalidation event, queue a `nBullMQ`n cache-warm job for the top 20 most-searched city/category/radius combinations.`n`n`nWarm jobs run at off-peak times (02:00–05:00 local) to repopulate expired popular searches.`n`n`n`n`n3.7 Navigation`n`n`nAdd 'Open in Google Maps' button on every business profile.`n`n`nLink format: https://www.google.com/maps?q=LAT,LNG`n`n`nNo in-app navigation. No Routes API. No Directions API.`n`n`n`n`n3.8 Duplicate Business Detection`n`nFlag as duplicate only when 2 or more of these signals match simultaneously:`n`n`n`nSignal`n`nRule`n`nGPS coordinates`n`nWithin 10 m of an existing listing`n`nBusiness name`n`n80%+ similarity — use `npg_trgm`n. Normalize before comparison: lowercase, trim spaces, strip punctuation, normalize Unicode, apply transliteration.`n`nPhone number`n`nExact match — normalize to E.164 format before comparison.`n`nAddress text`n`n85%+ similarity — use `npg_trgm`n after same normalization as name.`n`n`n`n`n1 signal match = soft warning, user can continue.`n`n`n2+ signals = confirmation prompt shown to user.`n`n`nNever hard-block automatically — always allow user override.`n`n`nValidate on both frontend and backend.`n`n`n`n`n`npg_trgm`n`n is already available in PostgreSQL with `nPostGIS`n setups. It is free, fast, and handles fuzzy text matching natively — no extra library needed. Enable with: CREATE EXTENSION IF NOT EXISTS `npg_trgm`n;`n`n`n`n`n3.9 Provider Abstraction Layer`n`n`nNever call Google API clients directly in application code.`n`n`nWrap all map/location API calls in three `nNestJS`n injectable services: `nGeocoderService`n, `nPlacesService`n, `n`nMapProvider`n`n.`n`n`nThe rest of the app calls only these services. Switching providers later requires changing only these files.`n`n`nAdd a fourth service: `nSearchLocationService`n — this is the single entry point for all combined `nElasticsearch`n + `nPostGIS`n search queries. No other layer calls `nElasticsearch`n or `nPostGIS`n directly for search.`n`n`n`n`n3.10 Cost Controls`n`n`nSet hard daily quota limits for all paid APIs in Google Cloud Console.`n`n`nSet billing alerts at 50% and 90% of monthly budget.`n`n`n`nImportant: Business profile pages must remain publicly accessible (no login required to view them). The Google Maps Embed API is free for public pages. Putting it behind a login wall may violate Google's Terms of Service.`n`n`n`n`n3.11 `nOpenStreetMap`n Tile Fair-Use Policy (Mobile Fallback Only)`n`n`n`nNEW in v2.4: OSM tile fair-use rules added. Violating these terms can result in your app being IP-blocked by OSM tile servers — causing the fallback to fail for all users.`n`n`n`nThe OSM static tile fallback is used only in the mobile app, only when the Google Maps SDK fails, and only for a single static tile per failure event. Despite this limited use, the following rules apply:`n`n`n`nRule`n`nDetail`n`nTile server`n`nUse tile.openstreetmap.org for MVP only. If monthly active users exceed 50,000, migrate to a self-hosted tile server or a CDN-backed provider (e.g., Stadia Maps free tier, `nProtomaps`n).`n`nUsage cap`n`nThe OSM Tile Usage Policy prohibits bulk or automated tile fetching. This fallback fires only on SDK failure — one tile per event. This is compliant.`n`nCaching`n`nCache returned OSM tiles on the device for a minimum of 24 hours. Never re-fetch the same tile within 24 hours.`n`nAttribution`n`nDisplay "© `nOpenStreetMap`n contributors" text on any screen showing an OSM tile. This is legally mandatory under the `nODbL`n `nlicence`n.`n`nMigration trigger`n`nAt 50,000+ MAU: migrate OSM tiles to Stadia Maps free tier (no key required up to 200,000 tile requests/month) or self-hosted. Document this in post-launch roadmap.`n`n`n`n`n4.
